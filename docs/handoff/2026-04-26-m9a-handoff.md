## Correctness fixes to ship first (in build order)

### Fix 1: Postgres connection leak in `atlas_dashboard.py`

**Problem:** Several fetcher functions open a `_pg_conn()` but only call `conn.close()` in the happy path. An exception leaves the connection open. Over time this exhausts the Postgres connection pool.

**Where:** `orchestrator/atlas_dashboard.py`: every function that calls `_pg_conn()`.

**Functions that need `conn.close()` in a `finally` block:**
- `_spend_aggregates()`
- `_spend_7d_by_day()`
- `_last_autonomous_action()`
- `_router_log_fallbacks()`
- `_error_log_tail()`
- `get_cost_ledger()`

**Pattern to apply:**
```python
conn = _pg_conn()
try:
    # existing code
finally:
    conn.close()
```

Read the file first to find the exact structure before editing.

### Fix 3: Structured JSON return schema in `run_chat()`

**Problem:** `run_chat()` currently returns a plain string. The M9 action-button system requires a structured response so the caller knows whether to send buttons.

**Where:** `orchestrator/handlers_chat.py`: `run_chat()` return value.

**Change:** Wrap the return in the M9 schema:
```python
{"reply": "...", "actions": [...], "artifact": {...}}
```

Parse the model's output with `json.loads()` in a `try/except`. On parse failure, return `{"reply": raw_content, "actions": []}`. Never raise. Log the parse failure.

Strip `actions` from the text before saving to Postgres conversation history (tool call messages and button arrays must not pollute the text log).

### Fix 4: Double-send bug in `handlers.py`

**Problem:** The current `handlers.py` chat dispatch calls both `send_message_with_buttons` AND `send_message` in some paths, causing Boubacar to receive two messages for one turn.

**Where:** `orchestrator/handlers.py` lines ~210-214 (read the file to find the exact lines: they may have shifted).

**Fix:** Replace those lines with a single call to `run_chat_with_buttons(message, session_key, chat_id, channel="telegram")`.

**Add this wrapper to `handlers_chat.py`:**
```python
def run_chat_with_buttons(message, session_key, chat_id, channel="telegram"):
    result = run_chat(message, session_key)
    if result.get("actions"):
        send_message_with_buttons(chat_id, result["reply"], result["actions"])
    else:
        send_message(chat_id, result["reply"])
```

This ensures exactly one message is sent per turn, regardless of whether there are buttons.

## Telegram action buttons feature (build after all 6 fixes above pass)

### What to build

Upgrade the **approval queue notification** so it arrives in Telegram with `[Approve]` and `[Reject]` inline keyboard buttons directly on the message. Today Boubacar has to type `yes confirm` or `no` in reply. After M9a, he taps a button.

This is NOT a new notification. It is an upgrade to the existing notification that already fires when a new approval queue item arrives.

### Where the existing notification lives

Read `orchestrator/notifier.py` or `orchestrator/publish_brief.py`: find where approval queue Telegram messages are sent. The function `send_message_with_buttons` in `notifier.py` already exists and already sends inline keyboard buttons. The approval notification just is not using it yet.

### What to add to `handlers_approvals.py`

Add a `_build_button(label, callback_data)` helper:
```python
def _build_button(label: str, callback_data: str) -> dict:
    assert len(callback_data.encode()) <= 64, f"callback_data too long: {callback_data!r}"
    return {"text": label, "callback_data": callback_data}
```

The 64-byte limit is a hard Telegram constraint. The assert catches violations at build time, not at 2am when a message fails to send.

Add callback dispatch cases in the existing callback handler for whatever new button prefixes you define (e.g. `approve_queue_item:`, `reject_queue_item:`). Mirror the existing `yes confirm` / `no` handler logic: do not duplicate the Notion write code, just route to the same underlying functions.

All of these must be true before marking M9a complete and deploying:
