# deliverables/ - Client-Facing Deliverable Artifacts

Production-quality artifacts that go to clients or prospects. Distinct from `outputs/` (raw agent output) and `workspace/` (in-progress).

## What lives here

| Subfolder | Purpose |
| --- | --- |
| `teardowns/` | Website teardown reports (e.g. `catalystworks-consulting/` self-teardown 2026-05-01) |
| Per-client deliverable folders (when shipped) | Final reports, audit one-pagers, polished assets |

## What does NOT live here

- Raw or in-progress agent output (`outputs/` / `agent_outputs/`)
- Client engagement working files (`workspace/clients/<slug>/`)
- Internal-viability reports (those stay internal in `outputs/` or workspace)
- Code or skills

## Rules

- A file lands here only when it has been (a) reviewed by Boubacar OR (b) generated by an audit-pass that explicitly classified it as client-facing.
- Internal viability framing must NEVER leak into deliverables here. See website-teardown skill for the dual-output pattern (internal-viability-report vs client-teardown-report).
- Em-dash hard rule applies: scrub before any send.
